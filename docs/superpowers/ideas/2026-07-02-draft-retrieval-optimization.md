# Brainstorm seed — A shared hierarchical document index (fixes draft retrieval AND understand sampling)

**Status:** pre-brainstorm brief. This is NOT an approved design. A future session should run the `superpowers:brainstorming` skill from here (issue → options×criteria → recommendation, one decision at a time; best-practice-first, verify via research) and only then write a spec.

**Framing (updated 2026-07-03):** what started as "draft re-runs retrieval per section" turns out to be one half of a bigger, single design. The *same* structure — a hierarchical index of each large document, built once — solves two currently-separate problems: (1) the Draft stage re-retrieves per section (below), and (2) the Understand stage samples large docs with a front-biased prefix (see "Second consumer"). Build the index once; **understand reads its top (a global query), draft reads its leaves (a local query).** The rest of this brief carries both.

## The problem (quantified on the Chevron run)

The Draft stage (and the draft-verify judge) **re-run retrieval independently for every section over the same large document**. On the Chevron 10-K run:

- 26 of 30 draft sections are anchored by the one 126-page 10-K.
- Each drafter's first `draft-read` returns the **126-page overview directory** (~4–5K tokens); ~26 drafters re-read it → **~100–130K tokens spent re-reading the table of contents alone.**
- Each drafter then **re-reads overlapping page content**: the 16 MD&A subsections all escalate into pages ~36–47; pages 36/39/40 were fetched by nearly every MD&A drafter.
- Result: the 16 MD&A leaves cost **~770K tokens** to draft ~12 pages of source. Draft total was 1.70M tokens; verify added 0.99M — the bulk carries this redundancy.

**Root cause:** the Draft stage **couples retrieval (find + read the right pages) into generation (draft the claims), per section, with no cross-section reuse.** The existing tiered read (`build_read`: overview→escalate) optimizes a *single* doc read (avoids dumping 599K chars) but there is no shared page→section index and no shared cache across drafters. This is the classic RAG anti-pattern: re-retrieving for every generation instead of retrieving once and generating many.

See `chevron/benchmark/CONCLUSIONS.md` (Finding 4) for the full data. Note this is a *cost/latency* problem only — quality was excellent (547 claims, 0 contradictions); do not trade away the per-section attribution or the honest-abstention behavior to fix it.

## Second consumer — the Understand-stage sampling problem

`tlddr/understand/slice.py::build_slice` hands the comprehension agent a **front-biased prefix**, not a representative view: the "content sample" is `doc.content[:8000]` (pure prefix truncation) and the "structure" section is the first 60 markers in document order (`structure[:60]`). For the HTML-extracted 10-K this degrades further — the extractor deliberately does not infer headings (design D3), so `_MARKER` matches only `--- page N ---`, and the structure skim becomes 60 **bare page numbers** with no semantic hint. Net: for the 126-page 10-K the agent's comprehension rested on the first ~8K chars (cover + start of business description) plus its own general 10-K knowledge — it never sampled pages 40–126.

This is *why* understand cost is flat ~22K/doc (the sample is hard-capped, so size stops mattering — benchmark Finding 1), and it is a real fidelity limitation for section-tagging on large docs (a front prefix can miss a topic that only appears late with no early signal; the Chevron coverage map leaned partly on 10-K-type priors, not on sampled tail evidence). Best practice for representing a long doc for a routing/comprehension decision is **structure-first** (present the document's own skeleton + a small sample per section) rather than a raw prefix — which is exactly the top of the hierarchical index this brief proposes. So the same index that de-duplicates draft retrieval also gives understand a representative, structure-aware sample at ~the same cost.

## Options already surfaced (starting point for the brainstorm, not a decision)

| Option | What it does | Redundancy removed | Cost / trade-off |
|---|---|---|---|
| **A. Cluster sections by shared evidence** | One drafter per *evidence region* instead of per section (e.g. all 16 MD&A subsections drafted by one subagent that reads pages 36–47 once; the 7 statements as another). | Both the overview re-read AND the page-content re-read for clustered sections (est. ~70–75% off the MD&A cluster). | Loses per-section isolation + per-section benchmark granularity; larger single-subagent context. Make clustering configurable (cluster for production, one-per-section for benchmarking). |
| **B. Precompute a page→section index (a "plan" step)** | One pass maps the 10-K's pages to sections up front (could be cheap/deterministic from the extractor's own captured heading markers, or one LLM pass); each drafter is *handed* its exact pages, skipping overview+escalate discovery. | The 26× directory re-read and the escalation exploration (~100–130K+). Overlapping *content* reads remain. | Adds one indexing pass; keeps one-subagent-per-section isolation (preserves attribution + benchmark granularity). RAG-correct separation of retrieval from generation. |
| **C. Shared prompt-cache prefix** | Read the shared pages once into a cached context block all drafters share (Anthropic caching → 10% input cost on cache hits). | The *cost* of re-reading (not the reads themselves) — repeated content billed at 1/10th. | Needs the orchestration/harness to share a cached prefix across subagents; the current "fresh subagent per task" model does not do this natively. Later optimization, platform-dependent. |
| **D. Shared hierarchical index (the convergence)** | Build a per-doc **tree** once at understand time (section/summary nodes over raw page leaves) and serve it to both stages: understand reads the top nodes; draft routes to a section's subtree and reads its leaves. Generalizes B (the page→section index becomes the tree). | Both problems at once: the 26× draft re-retrieval AND the front-biased understand sample. | The most structural change; needs the index contract + a router. But see the navigation section — for tl-ddr the router is *deterministic* (no embeddings/vector store), so the added machinery is small. |

**Provisional lean (to be validated in the brainstorm, not adopted here):** **D** is now the front-runner because it fixes both consumers with one artifact and, crucially, tl-ddr can build the deterministic variant with **no new heavyweight deps** (see navigation section — the section→page map already exists from understand). B is D-minus-the-understand-half (a fine smaller first step). A (clustering) stacks on top for dense regions as a configurable production mode. C depends on a harness capability the runtime doesn't expose. Irreducible floor: two sections that both genuinely need page 36 will each consume it once — that's fine; the fat is the ~26× directory re-read and the redundant escalation.

## Best-practice navigation methods (researched 2026-07-03; verify again at brainstorm time)

Building the tree is one thing; **navigating it for generation** is a separate, mature body of named methods. The brainstorm should adopt from these rather than invent:

- **RAPTOR — tree traversal vs collapsed tree** ([arxiv 2401.18059](https://arxiv.org/html/2401.18059v1)). Two ways to query a summary tree: *tree traversal* (root → top-k → descend, rigid top-down) vs *collapsed tree* (flatten all levels into one pool, retrieve across levels under a token budget). The paper found **collapsed tree consistently wins** — lesson: **don't force strict level-by-level drill-down; let the query see summary and leaves together.** For tl-ddr: hand the drafter its section-node summary (context) *and* the candidate leaf pages (evidence) at once, replacing the current overview→escalate round-trip.
- **LlamaIndex productized patterns** — *recursive/routed retriever* (retrieve an index node, follow its pointer to the underlying retriever for that subtree — this is exactly "route to the section, then pull its pages"); *auto-merging / small-to-big* (retrieve precise leaf chunks, merge up to the parent for fuller generation context); *document-summary index* (route on summaries first, retrieve within).
- **GraphRAG — local vs global search** ([From Local to Global, arxiv 2404.16130](https://arxiv.org/html/2404.16130v2)). *Global search* = map-reduce over community/summary nodes for whole-doc questions; *local search* = traverse the neighborhood of specific nodes for specific-fact questions (cheaper, more grounded). **This local/global split is exactly tl-ddr's understand/draft split**: understand = global (top nodes, "what is this doc"), draft = local (route to the section's pages, "the exact figures").

**The unifying principle:** *coarse-to-fine routing with generation grounded at the leaves* — route at summary level to find *where*, drill to leaves for *evidence*, generate against the leaves, and **cite the leaves, never summary nodes** (also how GraphRAG/RAPTOR limit hallucination; and it preserves tl-ddr's existing grounding invariant — citations resolve to raw `(node_id, page)`).

**The tl-ddr-specific advantage (why this is cheaper here than generic RAG):** tl-ddr **already knows the query's target** — you are drafting a *named section*, and understand already produced a deterministic **section → node → page** mapping (that is what `understand-retag` built). So the router is **deterministic, not embedding similarity** — no vector store, no embedding model, no new heavyweight deps. The tree only has to solve *within-document page selection* (which pages under this section hold the claim's evidence), a much smaller, structure-native problem than open-domain retrieval. Generic RAG must *infer* the subtree from a free-text query; tl-ddr routes by section id directly.

## Questions the brainstorm must settle

1. **Index build + contract.** Where does the index/tree live and who builds it — a deterministic `tlddr` step (from the document's own section structure) or a cheap LLM pass, or reuse of the existing understand-stage `section → node → page` tags? What is its persisted contract (`chevron/work/`-style artifact)? Prerequisite: the extractor must capture section/heading provenance for a structure-native tree (the HTML extractor's D3 skipped headings — that gap must close first, and it also fixes the understand skeleton).
2. **How deep a tree?** Structure-native (build the hierarchy from the document's own headings/Items — no deps, best for well-structured filings) vs RAPTOR-style embed-cluster-summarize (heavier deps; earns its keep only on unstructured text). For SEC filings the native TOC is high-signal — provisionally structure-native.
3. **Navigation method.** Collapsed-tree (summary + candidate leaves together) vs strict traversal; deterministic section-id router vs similarity. Provisional: deterministic router + collapsed-style presentation (summary + leaves at once). Grounding stays pinned to leaves.
4. **Understand consumer.** Replace `build_slice`'s prefix with a structure-first sample drawn from the tree top (global read) — measure section-tagging fidelity vs today's prefix on a large doc.
5. **Draft consumer / clustering.** Is clustering (A) a separate mode, or does the routed index (B/D) subsume the win? Measure index-alone vs index+clustering. How to keep per-section attribution + per-section benchmark granularity if sections are clustered (emit per-section claims from a clustered drafter?).
6. **Verify consumer.** Does the same structure help draft-verify (the judge already reads only cited pages, so its redundancy is per-section overhead, not exploration)?
7. **Regression guard.** The fix must not weaken grounding (every claim still resolves to a real raw page, never a summary node) or honest abstention.
